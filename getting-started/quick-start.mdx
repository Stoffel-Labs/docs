# Quick Start

This guide gets you from a fresh 0.1.0 installation to a working Stoffel computation. You will run the same tiny addition workflow two ways:

1. clear local execution for fast logic checks;
2. local MPC-style execution with secret inputs.

## Prerequisites

Finish [Installation](./installation) first. You should have:

- the `stoffel` CLI on your path;
- a local `StoffelVM` checkout;
- `target/debug/stoffel-run` built in that checkout for local MPC runs.

Check the CLI:

```bash
stoffel --help
```

## Option A: start from the CLI template

Create a project:

```bash
stoffel init hello-mpc
cd hello-mpc
```

The default template currently creates:

```text
hello-mpc/
├── Cargo.toml
├── README.md
├── Stoffel.toml
└── src/
    ├── main.rs
    ├── main.stfl
    └── stoffel_bindings.rs
```

Run the basic project lifecycle:

```bash
stoffel check
stoffel build
stoffel run
```

Useful follow-up commands:

```bash
# Print bytecode metadata before running.
stoffel run --program-info

# Disassemble generated bytecode.
stoffel compile --disassemble target/debug/hello-mpc.stflb

# Run local MPC once instead of entering watch mode.
stoffel dev \
  --runner /path/to/StoffelVM/target/debug/stoffel-run \
  --parties 5 \
  --threshold 1 \
  --once

```

Use the exact inputs expected by your generated `src/main.stfl`. Some templates use named function arguments, while ClientStore examples use `--client-input SLOT=VALUE`.

## Option B: run the Rust SDK quickstart

Create a new Rust binary project outside the repository:

```bash
cargo new stoffel-quickstart
cd stoffel-quickstart
```

Add dependencies in `Cargo.toml`:

```toml
[dependencies]
stoffel = { package = "stoffel-rust-sdk", git = "https://github.com/Stoffel-Labs/StoffelVM", tag = "v0.1.0" }
tokio = { version = "1.47.1", features = ["macros", "rt-multi-thread"] }
```

For development against a local checkout, use a path dependency:

```toml
[dependencies]
stoffel = { package = "stoffel-rust-sdk", path = "../StoffelVM/crates/stoffel-rust-sdk" }
tokio = { version = "1.47.1", features = ["macros", "rt-multi-thread"] }
```

Replace `src/main.rs` with a clear local execution check:

```rust
use stoffel::prelude::*;

fn main() -> stoffel::Result<()> {
    let result = Stoffel::compile("def main(a: int64, b: int64) -> int64:\n  return a + b")?
        .with_inputs(&[("a", 42_i64), ("b", 58_i64)])
        .execute_clear()?;

    println!("Result: {}", result[0]);
    Ok(())
}
```

Run it:

```bash
cargo run
```

Expected output:

```text
Result: 100
```

## Add local MPC execution

For the most reliable 0.1.0 local MPC smoke test, use `Share` inputs and open only the intended output:

```rust
use stoffel::prelude::*;

#[tokio::main]
async fn main() -> stoffel::Result<()> {
    let source = r#"
def add_private_values(a: Share, b: Share) -> int64:
  var sum = Share.add(a, b)
  return sum.open()
"#;

    match Stoffel::compile(source)?
        .parties(5)
        .threshold(1)
        .with_inputs(&[("a", 42_i64), ("b", 58_i64)])
        .execute_local_function("add_private_values")
        .await
    {
        Ok(result) => println!("Local MPC named-input result: {}", result[0]),
        Err(stoffel::Error::Unsupported(message)) if message.contains("stoffel-run") => {
            println!("Build stoffel-run first to execute local MPC: {message}");
        }
        Err(error) => return Err(error),
    }

    Ok(())
}
```

Run it:

```bash
cargo run
```

Expected output after the local runner starts and accepts masked inputs:

```text
Local MPC named-input result: 100
```

If the SDK cannot find the runner automatically, build it in your StoffelVM checkout:

```bash
cd /path/to/StoffelVM
cargo build -p stoffel-vm --bin stoffel-run
```

The `Share` and `ClientStore` examples are the recommended local MPC smoke tests because they exercise the local coordinator path.

## What changed between the two runs

In clear execution, Stoffel runs ordinary logic locally and returns the result directly. Use this for fast iteration.

In local MPC execution:

- values are represented as shares instead of ordinary clear application values;
- the source opens the computed result explicitly with `sum.open()`;
- the SDK configures party and threshold parameters;
- the local runner exercises the MPC-style path;
- the application works with the reconstructed result while the inputs remain modeled as shares during computation.

## Map product logic to MPC primitives

Good first Stoffel applications identify which values should be secret-shared, which values can stay public, and where reconstruction is required:

| Product goal | MPC shape to model |
| --- | --- |
| Eligibility checks | Secret inputs, public policy thresholds, reconstructed decision or score |
| Private set-style workflows | Client-provided shares, comparison/matching circuit, aggregate result |
| Private analytics | Secret-shared records, aggregate metric, explicit opening step |
| Threshold coordination | Parties, threshold, share generation, client-directed output |
| Personalization with less custody | Secret features, scoring function, reconstructed label or decision |
