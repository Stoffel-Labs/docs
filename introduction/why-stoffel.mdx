# Why Stoffel?

Secure multiparty computation (MPC) lets multiple parties compute over private inputs without first pooling those inputs in one place. The hard part for most developers is not understanding why that is useful; it is getting from a product idea to code that compiles, runs locally, and can be integrated into an application. Stoffel focuses on that gap.

## The MPC Development Challenge

### Traditional Approach Problems

**Cryptographic Complexity**
```
Developer → Learn MPC protocols → Implement low-level cryptography → Debug → Deploy
(6-12 months of specialized learning)
```

**Fragmented Ecosystem**
- Different protocols require different implementations
- No standard development tools or workflows
- Limited debugging and testing capabilities
- Deployment requires extensive infrastructure knowledge

**High Barrier to Entry**
- Requires specialized cryptography and distributed-systems knowledge
- Complex protocol implementation details
- Performance optimization requires deep understanding
- Integration with existing systems is difficult

### Real-World Development Pain Points

**Academic Research vs Production**
```
Research Paper: "Our protocol is 10x faster!"
Reality: No compiler, no debugger, no deployment tools, no documentation
```

**Tool Fragmentation**
```
Protocol A: Custom C++ library
Protocol B: Research Python scripts
Protocol C: Academic proof-of-concept
Result: No interoperability, no standard practices
```

## The Stoffel Solution

Stoffel packages MPC into a practical development experience: a language for computations, a compiler and VM, a project CLI, local MPC tooling, and a Rust SDK for application integration.

### 🎯 **Design Philosophy**

**Developer-First Approach**
- Familiar programming patterns and syntax
- Comprehensive tooling and development environment
- Clear interfaces between application logic and MPC protocol plumbing

**Practical MPC workflow**
- Write application logic in StoffelLang and compile it to VM bytecode
- Run clear local checks before exercising MPC-specific paths
- Use local MPC smoke tests and Rust SDK APIs to validate integration boundaries

**Protocol-aware, developer-facing**
- Exposes MPC concepts such as parties, thresholds, shares, and openings
- Keeps protocol plumbing behind a CLI, VM, and SDK interface
- Gives advanced developers lower-level hooks without making every app start there

## Key Advantages

### 1. **Familiar Development Experience**

**Modern Programming Language**
```stoffel
def secure_sum(a: Share, b: Share) -> int64:
  var total = a.add(b)
  return total.open()
```

**Comprehensive CLI**
```bash
# Current Stoffel 0.1.0 workflow
stoffel init secure-demo
stoffel check
stoffel build
stoffel run --local --parties 5 --threshold 1
```

### 2. **Complete Ecosystem**

**Integrated Components**
```
StoffelLang → StoffelVM → MPC Protocols
    ↓             ↓            ↓
Compilation   Execution   Network Layer
```

**Language SDK Integration**
```rust
use stoffel::prelude::*;

let result = Stoffel::compile("def main(a: int64, b: int64) -> int64:\n  return a + b")?
    .with_inputs(&[("a", 40_i64), ("b", 2_i64)])
    .execute_clear()?;
```

### 3. **Protocol Flexibility**

**Multiple Protocol Support**
- **HoneyBadger MPC**: Current primary protocol

- **Hybrid Approaches**: Mix protocols for optimal performance

**Configuration-Driven**
```toml

parties = 5
threshold = 1
curve = "bls12-381"
```

### 4. **Deployment-Oriented Building Blocks**

**Development Tools**
- Hot-reloading development server
- MPC simulation for local testing
- Comprehensive debugging capabilities
- Performance profiling and optimization

**Deployment planning surfaces**
- Network configuration builders
- Server/client handle construction
- Coordinator and on-chain integration surfaces
- Clear places to validate infrastructure, custody, and threat-model assumptions

## Technical Advantages

### Performance

**Optimized Implementation**
- Register-based VM for efficient computation
- Protocol-specific optimizations
- Hardware acceleration support
- Communication round minimization

**Workflow advantage**
```
Setup:       source install, CLI project, local runner
Development: clear logic checks plus local MPC-style smoke tests
Deployment:  explicit validation of protocol, SDK, infrastructure, and threat model
Maintenance: versioned bytecode and documented 0.1.0 APIs
```

### Security

**Security review posture**
- `mpc-protocols` has received an external audit from Zellic
- Other parts of the codebase have had AI-assisted security review, including tools such as veria.dev
- Deployment teams should validate the full stack, configuration, and threat model for their application
- Cryptographic best practices remain part of the implementation and review process

**Built-in security-oriented features**
- Explicit secret-share values and opening/reconstruction points
- MPC party and threshold configuration
- Network and coordinator surfaces for integration work
- Application-level access control and audit logging should be designed around your deployment

### Scalability

**Horizontal Scaling**

- Load balancing across computation nodes
- Geographic distribution support
- Auto-scaling based on demand

**Vertical Optimization**
- Efficient memory usage
- Parallel computation where possible
- Optimized networking stack

## The Bottom Line

Stoffel transforms MPC from a research curiosity into a practical development tool. It's the difference between:

- **Building your own database** vs **using PostgreSQL**
- **Implementing HTTP from scratch** vs **using a web framework**

Stoffel aims to be the approachable choice for building with MPC: technical enough to expose the right primitives, but packaged so developers can focus on application logic instead of protocol plumbing.
