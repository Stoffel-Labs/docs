Stoffel VM is the register-based runtime at the center of the Stoffel 0.1.0 stack. It lives in `crates/stoffel-vm` inside [`Stoffel-Labs/stoffel`](https://github.com/Stoffel-Labs/stoffel) and is consumed by the CLI, StoffelLang compiler, Rust SDK, and lower-level FFI surfaces.

## Workspace layout

| Crate | Purpose |
| --- | --- |
| `crates/stoffel-cli` | Cargo-like `stoffel` project CLI. |
| `crates/stoffel-bindgen` | Binding generation support. |
| `crates/stoffel-lang` | StoffelLang compiler. |
| `crates/stoffel-rust-sdk` | Primary Rust application SDK. |
| `crates/stoffel-vm` | Runtime, networking layer, MPC integrations, and C FFI. |
| `crates/stoffel-vm-types` | Shared value, instruction, function, and compiled bytecode types. |

## Execution model

```text
StoffelLang source (.stfl)
    ↓ compiler (stoffellang / CLI / SDK)
Compiled bytecode (.stflb)
    ↓ loader (stoffel-vm-types)
Program metadata + VMFunction definitions
    ↓ runtime (stoffel-vm)
Clear execution, local MPC execution, or network execution
    ↓
Allowed output values or client-directed MPC outputs
```

The VM is a register machine. It executes `Instruction` values from `stoffel-vm-types`, stores runtime values in register/table structures, and delegates MPC operations to registered builtin functions and backend engines.

## Runtime values

The current VM value model includes:

```rust
Value::I64(i64)
Value::I32(i32)
Value::I16(i16)
Value::I8(i8)
Value::U64(u64)
Value::U32(u32)
Value::U16(u16)
Value::U8(u8)
Value::Float(F64)
Value::Bool(bool)
Value::String(String)
Value::Object(usize)
Value::Array(usize)
Value::Foreign(usize)
Value::Closure(Arc<Closure>)
Value::Unit
Value::Share(ShareType, Vec<u8>)
```

Objects, arrays/lists, closures, foreign objects, and share objects are stored through VM-managed tables and references.

## Instruction families

The exact instruction enum is defined in `crates/stoffel-vm-types/src/instructions.rs` and must remain aligned with compiler code generation. Common families include:

- memory and movement: `LD`, `LDI`, `MOV`, `PUSHARG`
- arithmetic: `ADD`, `SUB`, `MUL`, `DIV`, `MOD`
- bitwise: `AND`, `OR`, `XOR`, `NOT`, `SHL`, `SHR`
- control flow: `JMP`, conditional jumps, `CMP`, `CALL`, `RET`
- object/list/closure support through runtime builtins and table operations
- MPC share operations through builtin calls such as `Share.add` and `Share.open`

## Builtins and MPC integration

`VirtualMachine::new()` registers the standard library and MPC-focused builtins. Important modules are:

- general runtime builtins: `print`, `type`, `to_string`, arrays/lists, objects, closures, `assert`, `slice`, and `contains`
- `Share.*`: clear-to-share conversion, arithmetic, randomness, opening, commitments, field/exponent helpers, and client output helpers
- `ClientStore.*`: client input slots for local or network MPC execution
- `Mpc.*`: runtime metadata, capabilities, and randomness helpers
- `MpcOutput.*`: coordinator/client output distribution
- lower-level `Rbc.*`, `Crypto.*`, `Bytes.*`, and `Avss.*` helpers

See [Built-in Functions](./builtins) for current names and examples.

## Bytecode format

Compiled programs are serialized as `.stflb` files through `stoffel-vm-types::compiled_binary::CompiledBinary`. The format carries function definitions, metadata, and enough manifest information for the CLI/SDK to inspect and execute programs.

```bash
stoffel compile src/main.stfl --output target/debug/main.stflb
stoffel compile --disassemble target/debug/main.stflb
```

## Embedding through the Rust SDK

The recommended application API is the Rust SDK:

```rust
use stoffel::prelude::*;

fn main() -> stoffel::Result<()> {
    let result = Stoffel::compile("def main(a: int64, b: int64) -> int64:\n  return a + b")?
        .with_inputs(&[("a", 40_i64), ("b", 2_i64)])
        .execute_clear()?;

    println!("Result: {}", result[0]);
    Ok(())
}
```

Local MPC execution from an application wrapper should normally load the bytecode built by the CLI:

```rust
use stoffel::prelude::*;

# async fn example() -> stoffel::Result<()> {
let result = Stoffel::load_file("target/debug/hello-mpc.stflb")?
    .parties(5)
    .threshold(1)
    .with_client_input(0, &[42_i64])
    .execute_local()
    .await?;
# Ok(())
# }
```

`.execute_local().await?` runs a local MPC test network on the developer machine. In deployment, MPC nodes run separately from input/output clients.

## Lower-level embedding

Advanced integrations can construct `VMFunction` values directly and execute them with `VirtualMachine`. This is useful for VM tests and bytecode-generation tools, but it is not the recommended first application path.

```rust
use std::collections::HashMap;

use stoffel_vm::core_types::Value;
use stoffel_vm::core_vm::VirtualMachine;
use stoffel_vm::functions::VMFunction;
use stoffel_vm::instructions::Instruction;

fn main() -> Result<(), String> {
    let mut vm = VirtualMachine::new();

    let hello = VMFunction::new(
        "hello".to_string(),
        vec![],
        vec![],
        None,
        2,
        vec![
            Instruction::LDI(0, Value::String("Hello from Stoffel VM".to_string())),
            Instruction::PUSHARG(0),
            Instruction::CALL("print".to_string()),
            Instruction::LDI(1, Value::Unit),
            Instruction::RET(1),
        ],
        HashMap::new(),
    );

    vm.try_register_function(hello)?;
    vm.execute("hello")?;
    Ok(())
}
```

## Status

The VM, compiler, CLI, and Rust SDK support local examples, template projects, `.stflb` inspection, and local MPC smoke tests. Use the CLI and Rust SDK for application workflows; use direct VM construction for VM tests, bytecode tooling, or maintainer-level integrations.
